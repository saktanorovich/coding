Couple of comments:

BagOfHolding:
10! is roughly 3 million. Each permutation requires a linear search to prove the item can either be taken out or not. Therefore, the total number of iterations is n*n!, which is about 30 million -- well within the time limits (see my practice room solution for how it can be done this way).

TeamBuilder (Div2):
The result may not have a row of all 1's for a node that can reach all other nodes. It may not have a path back to itself. The easy way to solve this is just to assume it has a path back to itself (i.e. set all G[i][i] = 1). Otherwise, you can avoid checking G[i][i] when making sure it has a path to all other nodes.

RaceCalculator (Div1):
You want the race with the lowest badness (the code is correct)

BagOfDevouring (Div1):

schveiguy
798 posts
I think I found an answer.

Form a huge cycle around all the sinks and sources. For example, if I can get from source x1 to sink y1, but I can't get from sink y2 to y1, then use a path from sink y2 to x1. Then, work on y1, find a sink it can't get to. At the end, you have a cycle by adding all the edges to get the sinks to get to eachother (i.e., you finally can get from sink yn back to y1). While you are connecting these edges, you need to try and use sources which have not been joined to before, if any exist. To do this, we use max flow on a bipartite graph with edges between the sources and the sinks they can reach. The edges used in the max-flow will be used in the cycle, and to complete the cycle, we add edges from sinks to sources that were not connected using a max-flow path (this is confusing, but I think it's correct). This will add min(|source|, |sink|) edges. After this large cycle, you may have some sources that haven't been cyclized, or some sinks that aren't cyclized (but not both). For each of these, just pick one source or sink that applies to it, and create an edge from the sink to the source, giving the full max(|source|, |sink|) edges.

FYI, SnapDragon and I came up with an algorithm (and hence a proof of possibility) that will show that you can connect M sources and N sinks with max(M, N) backedges such that every path between the sources and sinks is on a cycle.

The subproblem is this: given a bipartite graph of with partitions INIT (of size M) and TERM (of size N), with all edges directed from INIT to TERM, what is the minimum number of back-edges needed to ensure that every edge is on a cycle.

The answer is, as hypothesized, max(M, N).

The trick is to strengthen the induction hypothesis: you can actually connect the whole thing into one strongly connected component, rather than just ensuring that every edge is on a cycle. I'll leave details of the proof as an exercise unless somebody desperately needs them.

Maybe schveiguy is talking about matching on the complement of the bipartite graph? Even that doesn't work -- a maximum matching doesn't necessarily connect everything, and some maximum matchings can't be extended to an optimal cycling.

Consider source/sink graph of size (4,5)
0-a 1-b 2-c 3-d 0-e 2-e

A valid maximum matching on the complement would be
a-1 b-0 c-3 d-2

Then there's no back-edge you can add from e which will connect the whole thing, or put both 0-e and 2-e on a cycle.

However, there is as expected a 5-edge solution:
a-1 b-2 c-3 d-0 e-0
(which contains a maximum matching, just a different one)

You might be able prove that minimal cycle-covers contain maximum matchings on the complement, but the converse isn't true -- so I'm pretty sure matching per se isn't the way to go here.

Here is an algorithm for matching source to sink vertexes in a way which is guaranteed to put every edge (and hence every vertex) on a cycle. This algorithm can be applied after the (NP-complete) step of finding minimal source and sink sets.

Input: A directed bipartite graph G with partitions A and B, all vertexes of degree at least one, and all edges directed from A to B.

(Note: if you want to put in the isolated vertices from TeamBuilding, you have to split them into a pair of initial and terminal representatives with an edge between them).

Output: A set F of exactly max(|A|, |B|) back-edges (directed B->A) such that G union B is strongly connected.

(Note: we're going for strong connectivity rather than the weaker cycle-covering required, since it turns out to be easier to prove inductively).

We will proceed by induction/recursion on the total number of vertexes (|A|+|B|):

Case 0: |A| = |B| = 1 (base case)

F = the only back-edge available.

Case 1: |A| > |B|

Construct a set A' by selecting for each b in B some a in A which is adjacent to b. If A' has less than |B| vertexes (since some elements of A may have been selected more than one time), add arbitrary remaining elements from A until |A'| = |B|.

Now, consider the full subgraph G' of G induced by (A',B).
Every vertex in G' has degree at least one, by construction, so we can apply the algorithm to G' to obtain a set of back-edges F', |F'| = |A'| = |B|. By induction hypothesis, G' union F' is strongly connected.

Finally, if we add to F' arbitrary back-edges to the remaining |A|-|B| vertices in A / A', we get a set F such that G union F is strongly connected. |F| = |A|-|B|+|B| = |A| as required.


Case 2: |B| > |A|

By duality from case 1.

Case 3: |A| = |B|

If every vertex has degree 1, then the graph is a perfect matching. In that case, we can trivially add |A| back-edges zig-zag style to build a single cycle.

Otherwise:
Let x be some vertex in B which is *not* adjacent to any degree-1 vertex in A. Such an x must exist since the at most |A|-1 degree-1 vertexes in A can be adjacent to at most |B|-1 vertexes in B.
Similarly, let y be some vertex in A which is *not* adjacent to any degree-1 vertex in B.

Let A' = A / {y}, B' = B / {x}, and G' be the full subgraph of G induced by (A', B'). Again, all vertexes in G' have degree at least one, by construction, so we can apply the algorithm to G' to obtain a set of back-edges F', |F'| = |A|-1 = |B|-1.

There must be at least edge in G from A' to x (otherwise x must be solely connected to y, which would violate selection of y). Likewise there is at least one edge in G from y to B'. Therefore, if we set F = F union { x->y }, then G union F is strongly connected (x and y are on a path from G' back to G'). |F| = |F'|+1 = |A| = |B|.


QED

p.s. looks cubic to me.

Turns out that algorithm was overly complicated... as I realized on my way home.

Instead, we can do it easily with a greedy approach -- it's hard to go wrong.

The basic operation is to add a back edge, then remove the two vertexes X->Y it joines (adding in all the transitive w->X->Y->z paths as potentially new forward edges w->z).

The greedy choice is simply to connect two vertexes which aren't already connected! If the graph is of size M+N, this will reduce it to size (M-1)+(N-1) by adding one back edge.

This fails when the graph has become the complete bipartite graph K(M,N). Once that happens, you can add *any* back-edge, and the graph will reduce to K(M-1,N-1).

This works until we reach K(1,N) (or equivalently K(M,1)). At that point, we simply add the N back edges, and we're done!

If you've been counting, you'll see that we strongly connected the entire graph with exactly max(M,N) backedges.

Now *that* was a bunch easier. :)
